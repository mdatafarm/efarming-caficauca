Interface Oracle Efarming

Interface de comunicacion entre Efarming y cooperativa de caficultores de santander.

Plan de trabajo

Conexion directa con VPN

Conexión directa por VPN a la base de datos y creación de las consultas SQL (Etapa en la que estamos)

Implementacion API

Nuestro equipo realizara un proyecto WEB API para conectar esas consultas con el OASIS (En esta etapa la conexión la seguimos realizando por VPN y nosotros nos encargamos del deploymet)

Instalacion proyecto en servidor

Entregamos un proyecto web para que ustedes lo instalen en un servidor de su compañía y sirva como puente entre el OASIS y nuestro aplicativo. (Este proyecto web es solo un puente para llamar procedimientos almacenados autorizados por ustedes, el correspondiente proyecto no tendrá acceso a ningún otro recurso de la cooperativa Cadefihuil, Las ventajas de esta implementación es la seguridad segmentada, en la próxima reunión podemos revisar más a fondo este aspecto y entrar en detalle con la explicación técnica e implementación.)

Modelo CONCEPTUAL

A countinuacion se muestra el modelo descrito en la implementacion Como modelo de adopcion para desarrollar en la implementacion.

  • Modelo
  • Controlador
  • Transformado
  • Repositorio SQL
  • Mapeador

Modelo

El modelo estara compuesto de entidades definidas en este proyecto y que brinden comunicacion transparente con los demas entornos de comunicacion del EFarming

Las entidades seran creadas en la carpeta de modelo del proyecto

Estas entidades seran mapeadas con las colecciones de respuesta necesarias para la integracion

Para mayor informacion y entender el modelo de entidades puede referirce al siguiente TUTORIAL

Controlador API

En el controlador API ira toda la logica y los servicios de respuesta expuestos como servicios REST

Esta capa estara orientada a conectara las campas de BLL encargadas de hacer la traduccion de las peticiones

Para concer el proceso y entender a fondo el funcionamiento de web api se recomienda el siguiente TUTORIAL

Transformador

Titulado asi por simple gusto, no se le atribulle a ningun paradigma de arquitectura solamente un concepto para la implementacion

El transformador tendra como objetivo implementar las consultas SQL por medio de los componentes necesarios para ejecutar sentencias sql con Oracle

Este repositorio esta compuesto de las librerias y paquetes de Oracle necesarios para realizar el cast y llamado de la informacion provista por el tercero

Para conocer mas sobre esta implementacion y las buenas practicas de implementacion puede visitar el siguiente link

Tutorial de Oracle con .Net

Repositorio SQL

En este repositorio se almacenan las consultas SQL a ser utilizadas en la implementacion

Su finalidad es la de tener centralizadas las consultas y que puedan ser monitorizadas por la compañia a la que le prestamos el servicio

Esto proporciona un factor determinante en la seguridad del sistema al ser transparente en la ejecucion de consulta con nuestros clientes

Para concer un poco mas sobre como administrar un repositorio de consultas puede consultar el siguiente tutorial TUTORIAL

Mapeador

Realiza el proceso de mapear las respuestas con las colecciones de entidades
